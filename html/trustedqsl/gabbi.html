Trusted Gabbi (Global Amateur Interchange)  Formats
Sept. 04, 2001

Discussion:


There are currently many different interchange formats that are being used by amateur radio today. By far the most popular isADIF (Amateur Data Interchange Format)  for good reason. It is a very elegant, robust, and human readable. It has many of the features found in XML without the need for heavy weight libraries and the associated learning curve. Having said that ADIF does have a short comings. It doesn't have any concept of record type. Discussions on the ADIF development reflector hasn't been fruitful. Therefore to meet the requires of this TrustedQSL/LoTW projects the standard must be forked. The new standard will be called Gabbi (Global Amateur Interchange) . It is hoped that the two standards will be merged at some point in the future. Therefore a design goal is for Gabbi format to be backward compatible with ADIF. This turns out to be fairly easy to do by declaring the QSO/Logbook record as the default record type.  The QSO record type will use the standard already defined by ADIF.

Also see TrustedQSL High Level Virtual Record Design Notes by Michael Keane K1MK.   This is a very good conceptual of digital signing canonical form of TrustedQSLs.

Requirements:

  • Record Type Field. This field will determine the record type
  • An algorithm for determining the signing order of the fields must conform to canonical parse rules.  
  • In addition the currently define QSO/Logbook record, QSL and Station Records must be defined.
  • Field definitions is only significant to that record.  For example a GRID_SQUARE field on a QSO would refer to the contact station grid square.  GRID_SQUARE on a station record would refer to the grid square of the station submitting the QSL
  • A new field or possibly as well define virtual field (ie: call and number of seconds since epoch) which can be common to all records and required for tCONTACT records will be the GUID (GlobalUniqueIDentifier).  This is in order that different records can be uniquely connected to other records in a portable and database independent way.
  • C/C++ functions that parse and create record strings.

 

New Terms:

tCONTACT - Trusted Contact Record.  This record contains on the information about the contact and a pointer to the tSTATION record.

tQSL - A Trusted QSL is a match set of a tSTATION record and a tCONTACT record.  Many tCONTACT records can reference a single tSTATION record. 

tSTATION - Trusted Station Record.  This record contains information about the station used to make contacts.

 

Record Types:

Record Name Description
QSO This is also the standard logbook format.  To maintain backward compatibility with ADIF this is the default record type.  For details of this QSO Record type seehttp:://www.hosenose.com/adif for details
tCONTACT

This record will contain conformation information to be sent to the party the user had a QSO with along with the STATION record.

tSTATION

This record will contain conformation information about the station that a QSO was made from. 

Common Fields:

These are field that is common to all record types.  These are fields common both in name and mean to more than one record type.

Name Type Required Signed Max Size (bytes) Description
REC_TYPE C NZ Y 15 This defines the type of record.  The default is QSO record
SIGN C N Y 80 This is the digital signature of this record. Max size based on hex encoding
GUID C N Y 20 Global Unique IDentifier.  Exact description is TBD, but should be something like callsign and date/time group.  This maybe a virtual field.

tCONTACT (Trusted Contact) Record:

The tCONTACT record contains only the information that is unique to that contact.  Note that most of the fields are required.  This is because most of the fields are common to every contact.

Name Type Required Signed Max Size (Bytes) Description
BAND C N Y 10 The call sign of station contacted.
CALL C Y Y 20 The call sign of station contacted.
CERT_GUID C N Y 20 Digital Certificate GUID.  This would be CA DN & cert serial number. Defaults to the tSTATION GUID
DTG D Y Y 8 Date Time Group - In DD-MM-YYYY HH:MM:SS Format
FREQ N N Y 8 in Megahertz.  If unknown use Frequency from ADIF table for the band.
MODE C N Y 10 SSB, CW, RTTY, TOR= AMTOR, PKT, AM,FM, SSTV, ATV, PAC= PACTOR,CLO=CLOVER Defaults to the mode defined in tSTATION record
REC_TYPE C Y Y 10 Always "tCONTACT".  This declares the record to be a Trusted Contact record.
RST_SENT N Y Y 3 Signal Report Sent
EXCHANGE_SENT N N Y 6 Serial number of the event.
SIGN C Y Y 80 This is the digital signature of this record.  Base64 encoded
STATION_GUID C Y Y 20 This is the Station Record GUID (pointer) for the station that was used to make this contact.

  • Characters and Encoding

      • The "character set" shall be ISO/IEC 10646-1:2000 UCS ("Unicode v3.0")

      • Default encoding of characters shall be UTF-8

      • Optionally support UTF-16 encoding (TBR)

        • A GAbbI file object that is encoded in UTF-16 must begin with a two octet signature of #xFEFF (Zero Width No-Break Space; Byte Order Mark)

        • All implementations must check for the BOM signature. If the BOM signature is encountered indicating that the following file object is UTF-16 encoded and the reader cannot process UTF-16, the reader shall generate a fatal error

        • Interoperability is not guaranteed when using UTF-16

        • Support for other encodings (JIS, ISO 8859-1, ...)? (TBD) Would require a preamble or header

      1. Field Names

      • Reserve all field names beginning with the character sequence "gabbi" for use by the specification

      • Any other restrictions on what would be permissible field names? (TBD)

      1. User-Defined Fields

      • Default signing for all user-defined fields shall be unsigned

      • Unsigned default can be overridden for an individual field by means of the field type indicator

      1. Record Type

      • Add Header as a new REC_TYPE

        • Declares an optional GAbbI header record

        • Gathering "header" information in a GAbbI/ADIF compatible record permits multiple records being located within the ADIF header area (e.g. one Header record plus several tSTATION records)

      1. Field Type Indicator

      • Specify that presence of a field type indicator shall be treated as authoritative, overriding the default field type indicated by the specification

      • To permit overriding a field's default signing

        • Extend the ADIF field type indicator via an optional post-fixed "+" to indicate the field is signed (included in the virtual record) and a post-fixed "-" to indicate the field is unsigned

      • Expand the ADIF table of field type indicators to include

        • "O": octet stream, raw

        • "H": octet stream, hex encoded

        • "6": octet stream, base64 encoded

      1. Construction of the Virtual Record

      • Values shall be placed in the virtual record in the same order as its corresponding field name appears in a list of fields names sorted by increasing Unicode (from lower to higher values)

      • The virtual record shall contain no field delimiters (virtual record is write-only, is there ever a need to go back and parse it?)

      1. Digital Signature

      • The default digital signature (ds) algorithm shall be the Digital Signature Algorithm (DSA) of the Digital Signature Standard (FIPS 186-2); all implementations shall provided for signing and signature validations using DSA

      • Support for the ECDSA (ANSI X9.62) and the RSA ds algorithm (ANSI X9.31) may be provided. An indication of the signing algorithm used should be included in the GAbbI header. Interoperability is not guaranteed when using ECDSA or RSA ds.

      • Values (TBR) of the public parameters: p, q & g used in the DSA shall be provided in the specification.

      1. Forwards Compatibility and Default Behavior

      • Unrecognized Record Types should handled gracefully

      • Default behavior shall be to discard all information contained in any record of a type which the reader does not support (TBR)

      1. Procedures for Revising Specification

      • Typographical correction & other fixes

      • Consideration of amendments, modification & extensions

      • Version control & formal release of the specification

      1. Preferred Ordering of Fields within a Record

      • Is preference for field names sorted by Unicode (virtual record order)

      - OR -

      • Sorting by "significance" or "importance" (more human readable) (TBD)

      1. Preferred Ordering of Records

      • File object may and should start with a GAbbI unique "signature"

        • BOM as the first two octets indicates an UTF-16 encoded file to follow

        • TDB indicates GAbbI (UTF-8 default)

        • TDB for ASCII ?

      • ... may include a GAbbI header record

      • ... may and should place a GAbbI header record in the ADIF header area

      • ... may and should place tSTATION records in ADIF header area

      • ... may and should employ the ADIF end-of-header designator

      • ... may and should employ the GAbbI end-of-file designator to partition octet stream into "file objects"
